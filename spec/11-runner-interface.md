# Spec 11 — Runner Interface

> Status: **Draft** | Owner: — | Last updated: 2026-03-25
>
> Depends on: [Spec 10 — Host/Runtime Boundary](10-host-runtime-boundary.md)

## 1. Purpose

Define the single runner contract used by the host to invoke the runtime. This spec covers the `AgentRunner` interface, the `ExecutionRequest` and `ExecutionResult` shapes, execution origins, lifecycle, error handling, cancellation, and the `LocalRunner` reference implementation.

## 2. AgentRunner Interface

```js
/**
 * The contract every runner must implement.
 * The host calls execute() and receives a result.
 */
class AgentRunner {
  /**
   * Execute an agent task.
   *
   * @param {ExecutionRequest} request - Self-contained description of work.
   * @returns {Promise<ExecutionResult>} - Outcome of the execution.
   * @throws {RunnerUnavailableError} - Runner cannot accept work (e.g., shutting down).
   */
  async execute(request) {
    throw new Error('AgentRunner.execute() must be implemented');
  }

  /**
   * Cancel a running execution.
   *
   * @param {string} executionId - The ID from the ExecutionRequest.
   * @returns {Promise<boolean>} - true if cancellation was initiated, false if not found.
   */
  async cancel(executionId) {
    return false;
  }

  /**
   * Graceful shutdown. Reject new requests, wait for in-flight executions.
   *
   * @param {number} [timeoutMs=30000] - Max time to wait for in-flight work.
   * @returns {Promise<void>}
   */
  async shutdown(timeoutMs = 30_000) {
    // Default: no-op
  }
}
```

### 2.1 Method Contracts

| Method | Required | Blocking | Idempotent | Notes |
|--------|----------|----------|------------|-------|
| `execute(request)` | Yes | Yes (awaitable) | No | Each call produces a new execution. Duplicate `executionId` values should be rejected. |
| `cancel(executionId)` | No | Yes | Yes | Safe to call multiple times. Returns `false` if the execution already finished or was never started. |
| `shutdown(timeoutMs)` | No | Yes | Yes | After shutdown, `execute()` must throw `RunnerUnavailableError`. |

## 3. ExecutionRequest

A self-contained description of work the host wants the runtime to perform. The runtime must be able to execute using only the data in this object (plus its own injected dependencies like the LLM provider).

```js
{
  // --- Identity ---
  executionId: string,              // Unique ID for this execution (UUID v4, generated by host)
  origin: ExecutionOrigin,          // Why this execution was created (see §5)

  // --- Session context ---
  sessionId: string,                // Canonical session ID (e.g., "telegram:12345")
  userId: string,                   // Platform user ID
  channelId: string,                // Adapter channel ID
  userName: string | null,          // Display name (optional)
  sessionMetadata: object,          // Arbitrary session metadata from SessionManager

  // --- Conversation ---
  history: Message[],               // Conversation history (loaded by host)
  userContent: string,              // The new user message or task prompt

  // --- Tools ---
  toolSchemas: AnthropicToolSchema[], // Tool schemas available for this execution
  allowedToolNames: Set<string> | null, // Allowed tool names (null = all)

  // --- Prompt ---
  skillInstructions: string | null, // Skill instructions to inject (or null)
  memorySnippets: MemorySnippet[],  // Relevant memory search results (loaded by host)

  // --- Limits ---
  maxIterations: number,            // Max ReAct loop iterations
  timeoutMs: number | null,         // Overall execution timeout (null = no limit)

  // --- Timestamps ---
  createdAt: number,                // Unix ms when the request was created
}
```

### 3.1 MemorySnippet Shape

```js
{
  key: string,
  content: string,                  // Truncated to 300 chars by host
  metadata: object
}
```

### 3.2 Required vs Optional Fields

| Field | Required | Default | Notes |
|-------|----------|---------|-------|
| `executionId` | Yes | — | Host generates |
| `origin` | Yes | — | See §5 |
| `sessionId` | Yes | — | |
| `userId` | Yes | — | |
| `channelId` | Yes | — | |
| `userName` | No | `null` | |
| `sessionMetadata` | No | `{}` | |
| `history` | Yes | `[]` | Empty array for new sessions |
| `userContent` | Yes | — | |
| `toolSchemas` | Yes | `[]` | Empty if no tools available |
| `allowedToolNames` | No | `null` | `null` means all tools |
| `skillInstructions` | No | `null` | |
| `memorySnippets` | No | `[]` | |
| `maxIterations` | No | `25` | From config |
| `timeoutMs` | No | `null` | |
| `createdAt` | Yes | — | |

## 4. ExecutionResult

The outcome of a runtime execution. Returned by `runner.execute()`.

```js
{
  // --- Identity ---
  executionId: string,              // Echoed from the request

  // --- Outcome ---
  status: ExecutionStatus,          // See §4.1
  content: string,                  // Final text response (may be empty on error)

  // --- Messages for persistence ---
  newMessages: Message[],           // All messages generated during execution
                                    // (user echo, assistant turns, tool_use, tool_result)
                                    // Host persists these to conversation memory.

  // --- Metadata ---
  toolsUsed: string[],              // Names of tools invoked during execution
  tokenUsage: {
    inputTokens: number,
    outputTokens: number,
  },
  iterationCount: number,           // Number of ReAct loop iterations completed
  durationMs: number,               // Wall-clock execution time

  // --- Error (if status is not 'completed') ---
  error: {
    code: string,                   // Error taxonomy code (see §6)
    message: string,                // Human-readable error description
    retriable: boolean,             // Whether the host should consider retrying
  } | null,
}
```

### 4.1 ExecutionStatus

| Status | Meaning |
|--------|---------|
| `completed` | The runtime produced a final response. `content` is populated. `error` is `null`. |
| `max_iterations` | The runtime hit the iteration cap. `content` may contain a partial response. `error` is set. |
| `error` | A runtime or LLM error prevented completion. `content` may contain a fallback message. `error` is set. |
| `cancelled` | The execution was cancelled via `cancel()`. `content` may be empty. `error` is set. |
| `timeout` | The execution exceeded `timeoutMs`. `content` may be empty. `error` is set. |

## 5. Execution Origins

The `origin` field tells the runtime (and any observers) why this execution was created. The runtime may use this to adjust behavior (e.g., different fallback messages for scheduled tasks vs user messages).

```js
const ExecutionOrigin = {
  USER_MESSAGE: 'user_message',           // Direct user interaction via an adapter
  SCHEDULED_TASK: 'scheduled_task',       // Heartbeat or cron-triggered periodic task
  DELEGATED_AGENT: 'delegated_agent',     // Subtask dispatched by a parent agent or remote trigger
  MAINTENANCE_TASK: 'maintenance_task',   // Internal housekeeping (compaction, reindex, etc.)
};
```

### 5.1 Origin Behavior Expectations

| Origin | Session | Persistence | Delivery | Timeout |
|--------|---------|-------------|----------|---------|
| `user_message` | Real session from adapter | Full (history + new messages) | Outbound via adapter | Typically none (user waits) |
| `scheduled_task` | Synthetic `heartbeat:system` session | Optional (host decides) | Logged, not delivered to a user | Recommended (prevent runaway tasks) |
| `delegated_agent` | Inherited or synthetic | Host decides | Returned to caller | Recommended |
| `maintenance_task` | Synthetic | Typically not persisted | Logged only | Recommended |

## 6. Error Taxonomy

| Code | Meaning | Retriable | Typical Cause |
|------|---------|-----------|---------------|
| `runner_unavailable` | Runner rejected the request (shutting down, overloaded) | Yes (after delay) | `shutdown()` called, or runner at capacity |
| `runtime_error` | Unhandled exception in the runtime | Maybe | Bug in agent loop or prompt builder |
| `llm_error` | LLM provider returned an error | Yes (if transient) | API rate limit, server error, network issue |
| `tool_error` | A critical tool failure that halted execution | Maybe | Tool timeout, unexpected exception |
| `timeout` | Execution exceeded `timeoutMs` | Yes (with longer timeout) | Complex task, slow LLM responses |
| `cancelled` | Execution was cancelled | No | Host called `cancel()` |
| `policy_rejection` | Runtime detected a policy violation | No | Tool not in `allowedToolNames`, unexpected state |
| `max_iterations` | ReAct loop hit the cap | No (same input would loop again) | Complex multi-step task or loop |

## 7. Timeout and Cancellation

### 7.1 Timeout

- The host sets `timeoutMs` in the `ExecutionRequest`.
- The runner is responsible for enforcing the timeout. `LocalRunner` uses `Promise.race` with a timer.
- On timeout, the runner returns an `ExecutionResult` with `status: 'timeout'` and any partial results available.
- The timeout covers the entire execution, not individual LLM calls (those have their own retry logic in the provider).

### 7.2 Cancellation

- The host calls `runner.cancel(executionId)`.
- The runner sets an internal flag checked between ReAct loop iterations.
- Cancellation is cooperative, not preemptive. The current LLM call or tool execution completes before the cancellation takes effect.
- On cancellation, the runner returns an `ExecutionResult` with `status: 'cancelled'`.

### 7.3 Idempotency

- Each `executionId` must be unique. If the runner receives a duplicate `executionId`, it should reject with `runner_unavailable`.
- The runner does not store completed execution results. Idempotency at the application level is the host's responsibility (e.g., deduplicating adapter messages).

## 8. Streaming Compatibility (Future)

The initial `LocalRunner` returns buffered results only. The interface is designed to be forward-compatible with streaming:

```js
// Future extension — not required for LocalRunner
async *executeStream(request) {
  // Yields ExecutionEvent objects as the execution progresses
  // Final yield is the complete ExecutionResult
}
```

### 8.1 ExecutionEvent Shape (Reserved, Not Implemented)

```js
{
  executionId: string,
  type: 'text_delta' | 'tool_start' | 'tool_end' | 'iteration_start' | 'status_change',
  data: object,                     // Type-specific payload
  timestamp: number,
}
```

This extension does not change the synchronous `execute()` contract. Runners that support streaming implement `executeStream()` in addition to `execute()`.

## 9. LocalRunner Reference Behavior

The `LocalRunner` is the first and initially only implementation. It wraps the existing `AgentLoop`.

### 9.1 Construction

```js
class LocalRunner extends AgentRunner {
  constructor({ agentLoop, logger }) {
    super();
    this.agentLoop = agentLoop;
    this.logger = logger;
    this._active = new Map();       // executionId → { cancelled: boolean }
    this._shuttingDown = false;
  }
}
```

### 9.2 execute() Behavior

1. If `_shuttingDown`, throw `RunnerUnavailableError`.
2. If `executionId` already in `_active`, throw `RunnerUnavailableError`.
3. Register execution in `_active`.
4. Translate `ExecutionRequest` → internal format expected by `AgentLoop`.
5. Call `agentLoop.processMessage(translatedMessage)` with the request data.
6. Translate the `OutboundMessage` → `ExecutionResult`.
7. Remove execution from `_active`.
8. Return result.

### 9.3 Compatibility Expectations

The `LocalRunner` must produce identical outbound content for any given input compared to the current direct `AgentLoop.processMessage` path. Specifically:

- Same final text content for the same LLM responses.
- Same tool execution order and results.
- Same `newMessages` array for persistence.
- Same token usage accounting.

The only differences allowed are:
- Additional metadata fields in `ExecutionResult` that don't exist today (e.g., `iterationCount`, `durationMs`).
- The `executionId` field (new).
- Error shape changes (structured `error` object vs. thrown exceptions).

### 9.4 AgentLoop Modifications for LocalRunner

The `AgentLoop` will need minor modifications to support the runner contract:

1. Accept pre-loaded history and session metadata instead of loading them internally.
2. Return `newMessages` as part of the result instead of persisting them directly.
3. Stop emitting `message:outbound` directly (the host does this after the runner returns).
4. Accept cancellation signal checked between iterations.
5. Accept memory snippets and skill instructions as parameters instead of computing them.

These modifications are detailed in spec 14 (migration plan).

## 10. Design Decisions

| Decision | Rationale |
|----------|-----------|
| `execute()` returns a Promise, not a callback | Promises compose well with the existing async/await codebase. The host can await or race. |
| `ExecutionRequest` is self-contained | The runtime should not need to query external state. This enables future remote or containerized runners. |
| `newMessages` in the result | The host owns persistence. The runtime produces messages but doesn't write them. |
| Cooperative cancellation | Preemptive cancellation would require process isolation. Cooperative cancellation between iterations is sufficient for the local runner. |
| Error taxonomy as codes, not exception classes | Codes are serializable (future remote runners) and easy to match. The host can switch on `error.code`. |
| Streaming as a separate method | Adding streaming to `execute()` would change the return type. A separate `executeStream()` preserves backward compatibility. |
| `LocalRunner` wraps `AgentLoop`, doesn't replace it | Minimizes the blast radius of the first refactor step. The loop's internal logic is unchanged. |
